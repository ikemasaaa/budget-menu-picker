# UI改善計画（最終版）

この計画の判断基準は一貫している。  
このアプリは業務ツールではなく、「予算内で何が出るかを楽しむガチャ」である。したがって最優先は、管理画面のように情報を増やすことではなく、軽さを保ったまま「引きたくなる」「結果が出て気持ちいい」体験を作ることに置く。

## 議論サマリー

- 初版はフォーム改善の方向性自体は妥当だったが、体験の核であるガチャ演出の優先度が低すぎた。ここは Claude Code の指摘に同意し、最優先へ繰り上げる。
- 入力の複雑さは実際に課題であり、`src/main.ts` でも5本の制約行が初期表示される構成になっている。よって「予算中心の初期表示 + 栄養条件の折りたたみ」は採用する。
- アクセシビリティ修正は遊びのUIでも必須であり、44px級のタップ領域、24px以上のスライダーサム、コントラスト改善は優先度を上げる。
- 結果表示は「説明を増やす」より「価格と品目が一目で分かる」「出現の瞬間が気持ちいい」を優先する。初版の説明重視案は縮小し、軽い再設計に留める。
- アクセントカラー1色案には部分同意する。ただし独立した改善項目ではなく、結果出現時の演出に吸収する。
- 結果共有の重要性には同意する。ただし現状の共有は `X` 文言のみで、URLも結果再現も含まない。さらに `src/lib/url-state.ts` の `buildShareUrl()` は `tier` を含めていないため、そのままでは不十分。採用するなら「条件共有」ではなく「結果再現できる共有」まで含める。
- 「全チェーン混合ガチャ」は表現を修正したうえで条件付き採用とする。1回の結果に複数チェーンの商品を混ぜるのは不自然なので、「全チェーン横断で1チェーンを抽選してから通常のガチャを回す」方式に限定する。
- 初版のうち、固定CTA、条件要約の常時表示、予算消化率バー、2モード化、栄養ダッシュボード化は、このアプリの軽さに対して過剰と判断し見送る。

## 最終優先リスト

### Priority 1

#### 1. ガチャ演出を中核体験として作り直す
- 判断: 採用
- 内容: 押下直後に結果を即表示する現状をやめ、`300-600ms` 程度の短い溜めを入れる。ボタンは抽選中に `disabled` にし、結果カードはフェード + 微小移動で出す。品目は1品ずつ `80-120ms` 程度でスタガー表示する。
- 理由: 現状の `runGacha()` は同期的に検索してそのまま再描画しており、ガチャ体験として最も弱い。ここを直すだけで印象が大きく変わる。
- 実装メモ: `aria-live` の文言は `抽選中...` -> `結果が出ました` と切り替える。`prefers-reduced-motion` では即時表示へ落とす。表示後は結果セクションへスクロールし、見出しへフォーカスを移す。

#### 2. 制約入力を「基本」と「詳細」に分ける
- 判断: 採用
- 内容: 初期表示は `チェーン` `メニュータイプ` `予算` を中心にする。`カロリー` `タンパク質` `炭水化物` `塩分` は `詳細条件` に折りたたむ。必要なら `予算下限` も詳細側へ寄せ、初見では `予算上限` を主役にする。
- 理由: 今のUIは制約5行が最初からフルオープンで、カジュアル用途に対して威圧感が強い。2モード化ほど重くせず、1画面のまま認知負荷だけ下げるのが妥当。
- 実装メモ: `details` または簡素なアコーディオンで十分。ページ分割やステッパーは入れない。

#### 3. 基本アクセシビリティを先に直す
- 判断: 採用
- 内容: 入力・ボタンの高さを `44px` 以上へ寄せる。スライダーサムは `24px` 以上に拡大する。`--muted`、`reliability-badge`、チェーンバッジのコントラストを見直し、フォーカスリングを全操作要素で統一する。
- 理由: 現状の `src/styles.css` では補助色と一部バッジの視認性が弱く、スライダーサムも小さい。これはミニマルさとは別の問題で、早めに直すべき基礎品質である。

### Priority 2

#### 4. 結果カードを「一目で嬉しい」構成に整える
- 判断: 採用
- 内容: 合計金額を最上段で強く見せ、品目名と価格を今より読みやすく整理する。各品には小さく `kcal` を添える。合計栄養値は英語混じりの1行ではなく、日本語ラベルのコンパクトな一覧にする。
- 理由: 初版の説明強化案には一理あるが、このアプリで優先すべきは「理解」より先に「嬉しさ」である。結果カードは情報を増やしすぎず、視認性だけ上げるのがよい。
- 実装メモ: アクセントカラーは常時使わず、結果表示時の価格や強調ラベルに限定して使う。

#### 5. 共有を「結果再現できる共有」に引き上げる
- 判断: 採用
- 内容: `X` 共有にはテキストだけでなくURLを含める。URLには制約条件に加えて `tier` と結果識別子を含め、同じデータセット上では同じ結果を復元できるようにする。
- 理由: 今の共有は文言だけで、体験の拡散として弱い。しかもURL状態は条件の一部しか保持していないため、共有価値が中途半端である。
- 実装メモ: 結果識別子は `SearchResult.key` をベースにするのが現実的。データ更新で復元不能な場合は、条件だけ復元して「同一結果は再現不可」と表示するフォールバックを用意する。

#### 6. 候補なし時の再試行導線は最小限だけ足す
- 判断: 条件付き採用
- 内容: `lastResponse.diagnostics.suggestion` がある場合に限り、`予算を+100円して再抽選` のような単発アクションを1つだけ出す。
- 理由: Claude Code の「現在の診断で十分」という指摘には概ね同意する。ただし、すでに検索ロジック側に簡単な提案生成があるので、それをワンタップ化する程度なら過剰ではない。
- 実装メモ: 条件セクション全体と連動した複雑な救済UIにはしない。軽い逃げ道だけ用意する。

### Priority 3

#### 7. 全チェーン横断ガチャを追加する
- 判断: 条件付き採用
- 内容: `どこでもいい` をチェーン選択に追加し、その場合は「先にチェーンを抽選し、そのチェーン内で通常の組み合わせ探索を行う」。
- 理由: カジュアルアプリとして相性はよい。ただし現在の `searchMenus()` は `chainId` 前提で、1回の結果内に複数チェーンを混ぜる設計にもなっていない。面白さはあるが、コア改善より先ではない。
- 実装メモ: 結果カードには「今回は○○を引きました」と、選ばれたチェーンを明示する。

## 見送る項目

#### 1. 固定CTAと条件サマリーの常時表示
- 判断: 却下
- 理由: 詳細条件を折りたためばフォーム全体はかなり短くなる。モバイル固定CTAは結果セクションやフッターと競合しやすく、現在のミニマルさを崩す可能性が高い。

#### 2. スライダー現在値の別要約
- 判断: 却下
- 理由: 数値入力欄がすでにあり、情報が重複する。まずはタップしやすさと単位の分かりやすさを優先する。

#### 3. 予算消化率バー
- 判断: 却下
- 理由: このアプリでは `合計金額` と `おつり` で十分に伝わる。バーを足すとランダム提案より家計簿的な印象が強くなる。

#### 4. 条件不成立時の大がかりな条件連動UI
- 判断: 却下
- 理由: このアプリに必要なのは分析機能ではなく、もう一度回せる軽さである。単発の再試行提案までで止める。

#### 5. かんたん / 詳細の2モード化
- 判断: 却下
- 理由: 折りたたみで十分解決できる。モード分岐は状態管理も説明コストも増やす。

#### 6. 栄養カード・円環グラフ・栄養ダッシュボード化
- 判断: 却下
- 理由: 栄養管理アプリの方向に寄りすぎる。必要なのは確認しやすい数値であって、可視化の派手さではない。

#### 7. 結果履歴の比較UI
- 判断: 却下
- 理由: 面白さはあるが、今の弱点は比較機能ではなく「引いた瞬間の気持ちよさ」と「初見の入りやすさ」である。後回しが妥当。

#### 8. 結果カード内の詳細なデータ透明性表示
- 判断: 却下
- 理由: 免責と出典は現状のフッターで十分。結果カードには `※一部推定値` 程度の短い注意だけ残せばよい。

## 実装順の提案

1. `draw` 中間状態の導入、結果出現アニメーション、スクロール/フォーカス移動
2. 制約UIの折りたたみ、予算中心の初期表示
3. タップ領域、コントラスト、フォーカスリングの修正
4. 結果カードの軽い再設計
5. 共有URLの拡張と結果再現
6. 候補なし時の単発再試行アクション
7. 全チェーン横断ガチャ
