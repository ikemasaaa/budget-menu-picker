# 日本チェーン店の栄養データ取得戦略

作成日: 2026-03-31

## 0. 目的と前提

- 対象チェーン: 吉野家 / 松屋 / マクドナルド / スシロー
- 必要項目: 価格 / カロリー / たんぱく質
- 本書は調査・推論・設計のみ。実装は行わない。
- 一次情報を優先し、推測を含む箇所は `推測` と明記する。
- 現在わかっている環境制約:
  - Codex sandbox では外部 DNS 制約により `npm install` や `curl` が失敗しうる
  - リポジトリ内には Playwright 依存は存在するが、この sandbox でのオンライン収集は不安定
  - 一方で、Claude Code 実環境では Playwright とネット接続を前提に動かせる想定

## 1. 調査サマリ

結論から言うと、4チェーンを単一手法で揃えるのは悪手で、チェーンごとに最も安定した公開形態を使い分けるのが最適である。

- 吉野家: PDF が最も強い。価格は商品 HTML、栄養は PDF が安定。
- 松屋: 商品 HTML 自体に価格と栄養が載っており最も取りやすい。
- マクドナルド: 公式の栄養一覧 HTML テーブルが最良。少なくとも今回確認範囲では CSV は見当たらない。
- スシロー: 価格とカロリーは店舗選択後メニューで見えるが、店舗依存が強い。たんぱく質の公式一括公開は確認困難。

設計上は、`公式静的ソースを第一選択`、`Playwright は補助`、`非公式 API は原則不採用`、`スシローだけは人手確認前提の半自動運用` が現実的である。

## 2. チェーン別の公式公開状況

### 2.1 吉野家

確認できた事実:

- 公式メニュー一覧: https://www.yoshinoya.com/menu/
- 商品ページは HTML で公開されている。例: 豚丼 https://www.yoshinoya.com/menu/butadon/buta-don/
- 栄養・アレルギーは PDF でまとまって公開されている。
  - 確認できた PDF: https://www.yoshinoya.com/pdf/allergy/
  - PDF 冒頭に「メニュー別・栄養成分・アレルギー物質一覧」「メニュー全情報」とあり、商品別 kcal / g が掲載されている
  - PDF 内注記では「栄養成分値は、配合に基づいた推定値」と明記

評価:

- 栄養データの一次情報として PDF が使える。
- 商品ページ側で価格、PDF 側で栄養を突合する二段構えが現実的。
- CSV / JSON の公式公開は今回確認できなかった。

設計判断:

- 吉野家は `PDF + HTML 併用` を第一候補にする。
- PDF は表構造が比較的明快なので、OCR より PDF テキスト抽出ベースの方が品質が高い見込み。
- 商品名揺れとサイズ表現の正規化が主要コスト。

### 2.2 松屋

確認できた事実:

- 公式の栄養導線: https://www.matsuyafoods.co.jp/matsuya/service/nutrition.html
- 商品ページ HTML に価格と栄養が同居している。例: 牛めし https://www.matsuyafoods.co.jp/matsuya/menu/gyumeshi/gyumeshi_hp_250422.html
- 商品ページにはサイズ別の価格、カロリー、たんぱく質、脂質、炭水化物、食塩相当量がそのまま記載されている。
- 注記として「表示している特定原材料(アレルギー物質)・栄養成分は2026年3月31日現在のものです。」のような更新基準日が商品ページごとに見える。
- 一部商品では「沖縄店舗では…栄養成分一覧PDFをご確認ください」とあり、例外分岐も存在する。

評価:

- 4チェーン中もっとも収集しやすい。
- HTML だけで価格と栄養が揃うため、品質・保守性とも高い。
- 例外は地域差と店内/持ち帰り条件。牛めしのように「みそ汁込み」の注記があるため、その扱いをデータモデルで吸収する必要がある。

設計判断:

- 松屋は `商品 HTML 直接パース` が最適。
- 補助として service ページや PDF を使う程度で十分。
- 更新日時を自動保存しやすく、月次更新でも鮮度を維持しやすい。

### 2.3 マクドナルド

確認できた事実:

- 公式栄養一覧: https://www.mcdonalds.co.jp/quality/allergy_Nutrition/nutrient/
- 同ページは HTML テーブルとして商品名、エネルギー、たんぱく質などを掲載している。
- 商品名から各商品ページへリンクされている。
- 公式の栄養関連導線として「栄養バランスチェックの使い方」ページも存在する: https://www.mcdonalds.co.jp/sustainability/food/allergy_Nutrition/how_to/
- 今回の確認範囲では、この栄養一覧ページ上に CSV / PDF ダウンロード導線は見つからなかった。

評価:

- 少なくとも栄養は HTML テーブルで安定取得できる。
- 価格は別のメニュー商品ページや一覧から拾う必要があるため、栄養と価格のソース分離はある。
- API が仮に内部利用されていたとしても、公開 HTML だけでかなり足りる。

設計判断:

- マクドナルドは `公式栄養一覧 HTML + 商品ページ価格` の二段構えが妥当。
- CSV がない前提で設計して問題ない。
- 栄養ページは静的 HTML で読めるため、Playwright 必須ではない。

### 2.4 スシロー

確認できた事実:

- メニュー導線は店舗選択前提: https://akindo-sushiro.co.jp/apps/menu/
- 検索結果上でも「店舗を指定してメニューを表示」と明記されており、店舗別メニューである。
- 過去調査では商品詳細ページで価格と kcal は確認できたが、たんぱく質の公式確認は難しかった。
- 少なくとも今回の調査範囲では、店舗選択なしで全商品の価格・カロリー・たんぱく質を一括で見られる安定した公式ページは確認できなかった。
- スシローの利用規約ページは公開されている: https://www.akindo-sushiro.co.jp/about/

評価:

- 4チェーン中もっとも難しい。
- 価格が店舗別 / 地域別で変動しうるため、「単一の正解価格」を静的アプリに載せる設計と相性が悪い。
- たんぱく質の一次情報不足が大きい。アレルギー情報や商品詳細だけでは不足する可能性が高い。

設計判断:

- スシローは `完全自動化を狙わない` 方がよい。
- まずは代表店舗を固定して価格・カロリーを取得し、たんぱく質は手動確認または参考値扱いにするのが現実的。
- UI 上で「価格は代表店舗基準」「栄養は参考値を含む」表示が必須。

## 3. 番外: 代替データソースの実用性

### 3.1 文部科学省 食品成分データベース

評価:

- 食材単位の標準成分には強いが、チェーン店の完成品メニューを直接カバーしない。
- 牛丼チェーンやファストフードの実メニューの価格・商品名・期間限定品との対応付けには向かない。

結論:

- ベース推定や sanity check には使えるが、今回の主データソースには不向き。

### 3.2 カロリーSlism

確認できた事実:

- チェーン店名検索ではなく、一般料理・個別食品ページ中心のヒットが多かった。
- 検索結果例では「チーズ牛丼」の一般栄養説明ページが表示され、チェーン公式メニューとの 1:1 対応は弱い。

結論:

- 網羅性はあるが、チェーン公式メニューの正確な価格・商品定義を保てない。
- 参考値用途に留めるべき。

### 3.3 kalori.jp

確認できた事実:

- 松屋・スシロー等の商品ページが存在する。
- ページ上に「推定」「AI」表記があり、価格にも「実際の価格と異なる場合があります」と注記されるケースがある。

結論:

- 公式不足部分を埋める候補にはなるが、一次情報ではない。
- 特にスシローのたんぱく質補完には使えるが、UI で `推定値` と明示しない限り品質リスクが高い。

### 3.4 Open Food Facts

確認できた事実:

- 今回の検索基盤では有意な日本チェーン個別商品の coverage を確認できなかった。
- 検索エンジン上では Open Food Facts が robots により一部取得制限されている表示があった。

結論:

- 今回の対象チェーンには向かない。

## 4. 取得方法オプションの全評価

評価軸:

- 実現性: 今の環境で成立しやすいか
- 品質: 正確性・再現性・欠損の少なさ
- 保守コスト: 壊れにくさ、修正しやすさ
- 法的リスク: 利用規約・権利・過剰負荷の観点

### A. 公式 PDF / CSV 直接ダウンロード + パース

対象適性:

- 吉野家: 高
- 松屋: 中
- マクドナルド: 低〜中
- スシロー: 低

評価:

- 実現性: 高。ただし PDF の表抽出難易度はチェーン差あり。
- 品質: 高。一次情報に最も近い。
- 保守コスト: 中。レイアウト変更で壊れるが、HTML より大崩れしにくい場合がある。
- 法的リスク: 低〜中。公開資料の参照自体は比較的穏当だが、再配布や大量転載は避けるべき。

所見:

- 吉野家には最重要。
- CSV は現時点で有力な公開先を確認できていないため、`PDF があるなら PDF を主戦場にする` 発想が必要。

### B. Playwright によるヘッドレスブラウザスクレイピング

対象適性:

- 松屋: 中
- マクドナルド: 中
- 吉野家: 中
- スシロー: 高

評価:

- 実現性: Claude Code 実環境では高、Codex sandbox では低。
- 品質: 中〜高。JS 描画や店舗選択に対応できる。
- 保守コスト: 中〜高。UI 変更に弱い。
- 法的リスク: 中。頻度・負荷・規約次第。

所見:

- スシローではほぼ必須候補。
- ただし、HTML 静的取得で足りるチェーンにまで Playwright を常用すると、複雑さだけが増える。

### C. 食品データベース API / 外部 DB 活用

対象適性:

- スシローの不足補完: 中
- その他チェーンの主ソース: 低

評価:

- 実現性: 中
- 品質: 低〜中。公式との差分管理が難しい。
- 保守コスト: 低〜中。取得は簡単でも検証が大変。
- 法的リスク: ソース次第。ライセンス確認が必要。

所見:

- 主ソースには不向き。
- `公式値がない項目だけ補完` の限定利用なら意味がある。

### D. 手動メンテ + Google スプレッドシート管理

評価:

- 実現性: 非常に高い
- 品質: 中〜高。少量ならむしろ高品質
- 保守コスト: 中。項目数が増えるとつらい
- 法的リスク: 低

所見:

- 個人開発・4チェーン・静的配信という条件と非常に相性がよい。
- 特にスシローのような難物は、完全自動化より `手動レビュー付き半自動` が強い。

### E. GitHub Actions による CI 定期実行スクレイピング

評価:

- 実現性: 中
- 品質: 中
- 保守コスト: 中
- 法的リスク: 中

所見:

- 技術的には有用だが、個人開発で毎日回す意味は薄い。
- 月次や手動 dispatch に限定すれば価値がある。
- スクレイピング失敗時に壊れた JSON を main に流さない仕組みが前提。

### F. 公式アプリの API エンドポイント利用

確認できた事実 / 推測:

- 吉野家サイト上には公式アプリへのリンクとして AWS API Gateway ドメインが見える。
- 他チェーンもアプリやモバイルオーダー基盤を持つため、内部 API の存在自体は高確率。
- ただし今回、対象 4 チェーンで公開仕様や正式利用可能な API は確認できなかった。

評価:

- 実現性: 中
- 品質: 高い可能性はある
- 保守コスト: 非常に高い
- 法的リスク: 高

所見:

- 個人開発・非商用でも、最も避けるべき選択肢。
- 認証、署名、レート制限、アプリ更新で壊れる可能性が高い。
- 今回の用途では投資対効果が悪すぎる。

## 5. 推奨アーキテクチャ

### 5.1 基本方針

- `チェーンごとに取得戦略を分ける`
- `一次情報を主、推定値は補助`
- `完全自動化より、半自動 + 人間レビュー`
- `静的配信向けに、成果物は正規化 JSON と更新メタデータに落とす`

### 5.2 チェーン別の最適戦略

#### 吉野家

- 価格: 公式商品ページ HTML
- カロリー / たんぱく質: 公式 PDF
- 実装思想: 商品名正規化テーブルを持ち、PDF 行と商品ページを対応づける
- 更新頻度: 月次、または新メニュー投入時

#### 松屋

- 価格 / カロリー / たんぱく質: 公式商品ページ HTML
- 例外: 沖縄差分や店内みそ汁込み注記は metadata に保持
- 更新頻度: 月次

#### マクドナルド

- カロリー / たんぱく質: 公式栄養一覧 HTML
- 価格: 商品ページまたはメニュー一覧 HTML
- 期間限定商品が多いので、更新頻度は月次よりやや高めでもよい
- 更新頻度: 月2回 or 月次 + 新商品月だけ臨時

#### スシロー

- 価格 / カロリー: Playwright で代表店舗を選択して取得
- たんぱく質: 公式で一括確認できない場合は手動入力または参考値
- 代表店舗: 首都圏の通常店舗を 1 店固定
- 更新頻度: 月次。キャンペーン期間中は必要時のみ手動

### 5.3 データパイプライン構成

推奨ディレクトリ:

- `scripts/fetch-yoshinoya.ts`
- `scripts/fetch-matsuya.ts`
- `scripts/fetch-mcdonalds.ts`
- `scripts/fetch-sushiro.ts`
- `scripts/merge-menu-data.ts`
- `scripts/validate-menu-data.ts`
- `data/sources/raw/`
- `data/sources/normalized/`
- `data/menu-dataset.json`
- `data/menu-dataset.meta.json`

役割:

- `raw/`: 取得した HTML / PDF テキスト / Playwright dump
- `normalized/`: チェーン別の整形 JSON
- `menu-dataset.json`: アプリ本体が読む統合済み JSON
- `menu-dataset.meta.json`: 更新日、取得方法、代表店舗、参考値フラグ

### 5.4 UI に出すべき鮮度・品質表示

必須:

- 最終更新日
- チェーン別更新日
- データソース種別
  - 公式 HTML
  - 公式 PDF
  - 参考値
- スシローの代表店舗名
- 参考値混在の警告

表示例:

- `最終更新: 2026-03-31`
- `吉野家: 公式PDF/公式サイトをもとに更新`
- `スシロー: 東京都内の代表店舗価格。たんぱく質の一部は参考値を含みます`

## 6. 今すぐ実行できることの優先順位

### 6.1 Claude Code 実環境で今すぐやる価値が高いこと

1. 松屋の HTML 取得スクリプトを仕上げる
2. マクドナルドの栄養一覧 HTML パーサを作る
3. 吉野家 PDF の抽出精度を検証する
4. スシローの代表店舗固定 Playwright フローを確立する
5. 差分検知と手動レビューのワークフローを作る

理由:

- まず `自動化しやすい 3 チェーン` を固めると、プロダクト価値の大半が先に出る。
- スシローは最後に回してよい。

### 6.2 Codex sandbox 内でできること

- 既存 JSON / スクリプト構成の設計
- パーサのオフライン実装
- 正規化スキーマ設計
- バリデータ作成
- ドキュメント整備

### 6.3 Codex sandbox 内でできない / やりにくいこと

- 外部 DNS 前提のリアルタイム取得
- `npm install` 前提の追加依存導入
- `curl` などでの本番サイト確認
- Playwright のオンライン検証

### 6.4 人間が手でやった方が早いこと

- スシロー代表店舗を 1 店決める
- スシローのたんぱく質を「参考値で許容するか」を決める
- UI の注意書き文言を最終決定する
- 月次の差分レビューを 10 分で済ませる運用を決める

## 7. 法的・倫理的考慮

### 7.1 robots.txt

今回確認できたこと:

- Open Food Facts は検索結果上で robots 制限の影響が見えた。
- 対象 4 公式サイトの `robots.txt` 本文は、今回の検索基盤では直接確認しきれなかった。

実務判断:

- robots 未確認を理由に突っ走るべきではない。
- 利用規約と公開導線の明確さを優先し、`静的公開されている情報を低頻度・低負荷で取得する` 方針に寄せるべき。

### 7.2 利用規約・著作権

確認できた事実:

- スシロー利用規約では、サイト上情報の利用は非商業的かつ個人的利用を条件とし、複製・再配信・改変に強い制限がある。
- 松屋の Web 利用規約では、著作権法上の制約を超える使用を禁じる条項がある。
- 吉野家には規約関連ページが公開されている。
- マクドナルドもウェブサイト利用規約への導線を持つ。

実務判断:

- 元 HTML / PDF の再配布、画像転載、全文転載は避ける。
- アプリに載せるのは `抽出・要約した数値データ` と `出典リンク` に留める。
- 利用規約違反リスクを下げるには、クロール頻度を低くし、認証や隠し API に踏み込まないことが重要。

### 7.3 リスク水準の整理

- 公式 HTML / PDF から数値のみを低頻度で取得し、参考情報として表示:
  - リスクは `低〜中`
- Playwright で店舗選択後ページを人間同等の頻度で取得:
  - リスクは `中`
- 非公開 API の利用やアプリ通信の再現:
  - リスクは `高`

### 7.4 UI/README に入れるべき注意書き

- `本アプリは非公式です`
- `価格・栄養情報は各社公式公開情報をもとに更新していますが、最新の販売状況は各社サイト・店舗でご確認ください`
- `店舗差・時期差・持ち帰り条件により実際の数値や価格が異なる場合があります`

## 8. 最終提案

このプロジェクトに最適なのは、以下のハイブリッド構成である。

- 吉野家: 公式 PDF + 商品 HTML
- 松屋: 商品 HTML 直接取得
- マクドナルド: 栄養一覧 HTML + 商品価格 HTML
- スシロー: Playwright で代表店舗固定 + 手動レビュー

運用は `月次更新 + 必要時の臨時更新` で十分。GitHub Actions は自動定期実行より、まずは `手動 dispatch できる更新ジョブ` として入れる方がよい。理由は、静的サイト・個人開発・非商用という前提では、完全自動更新より `壊れたデータを出さないこと` の方が重要だからである。

最も重要な設計判断は 2 つある。

- 1つ目は、`スシローだけは自動化の完成度を他3社と揃えない` こと。
- 2つ目は、`推定値と公式値を同じ重みで見せない` こと。

この2点を守れば、品質・保守コスト・法的リスクのバランスが最もよい。
