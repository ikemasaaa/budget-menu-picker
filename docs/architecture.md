# Architecture

作成日: 2026-03-31

## 1. 技術スタック判断

### 結論

- UI方針: `Vite + React + TypeScript` が本来の最適解
- 配信方針: `静的配信`
- デプロイ先: `GitHub Pages`
- サーバー: `なし`

### 理由

#### Next.js ではなく Vite + React を選ぶ理由

- 今回の要件は検索条件入力と静的JSON探索が中心で、SSR・RSC・API Route の価値がほぼない。
- データ量は小さく、検索もブラウザ内で完結できるため、Next.js の複雑さはオーバースペック。
- 参考サービスも静的配信で成立しており、MVP の速度を優先するなら軽量SPAが妥当。

#### Vercel / Cloudflare Pages / GitHub Pages の比較

- Vercel: 将来APIやISRを使うなら強いが、今回は優位性が薄い。
- Cloudflare Pages: 高速だが、今回のMVPではGitHub連携以外の恩恵が限定的。
- GitHub Pages: 静的JSONとSPAを置くだけなら十分。参考事例とも整合する。

### API Route を用意しない理由

- 条件探索はクライアント側で十分高速。
- 価格・栄養データは静的JSONで管理できる。
- サーバーを持たない方が法務・運用・コストの面で軽い。

## 2. 実装上の現実対応

- この作業環境では外部npm依存を取得できないため、MVP本体は `Node 24標準のTypeScript実行` と `静的SPA` で実装する。
- 設計思想は Vite のシンプルさに寄せ、`src/` をブラウザ向けにビルドして `dist/` に出力する。
- 将来 `Vite + React` に移行しやすいよう、探索ロジック・URL状態・描画責務は分離する。

## 3. データ方針

- 対応チェーン: `サイゼリヤ / 吉野家 / 松屋 / マクドナルド`
- データソース方針:
  - サイゼリヤ: 既存OSSの公開JSON構成を参考に手動正規化
  - 吉野家・松屋・マクドナルド: 公式の価格・栄養公開情報を前提に手動整備した参考データ
- 各商品は `price / calories / protein` を必須にした単純スキーマとし、MVPでは検索速度と保守性を優先する。

## 4. 探索アルゴリズム

- 基本: 枝刈り付き DFS / Backtracking
- 制約:
  - 予算上限
  - カロリー上限
  - タンパク質下限
- ルール:
  - 同一商品は重複なし
  - 合計点数は最大5点まで
- 枝刈り:
  - 予算超過
  - カロリー超過
  - 残り候補をすべて使ってもタンパク質下限に届かない場合
- スコア:
  - 目的別プリセットに応じて `タンパク質量 / タンパク質効率 / 低カロリー性 / 予算消化率` を重み付け

## 5. MVPで入れるUX

- おすすめプリセット
- 条件に合わない場合の理由表示
- `予算 +100円` の条件緩和提案
- URLクエリ保存
- データ出典・更新日の表示
- スマホ優先のレスポンシブUI
