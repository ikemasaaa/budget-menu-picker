## 1. 類似サービスの調査

### 1-1. 近いサービス / ツール

#### 日本

1. サイゼリヤ1000円ガチャ
- URL: https://saizeriya-1000yen.marusho.io/
- 概要: サイゼリヤのメニューから 1000 円以内の組み合わせを出す非公式ツール。
- 特徴:
  - 予算条件が明快でネタ性が強い
  - 軽量で即結果が出る
  - PWA 化されておりオフライン寄りに動く
- 弱み:
  - 予算固定
  - 栄養条件を扱わない
  - 店舗横断で使えない

2. サイゼリヤN円ガチャ
- Qiita: https://qiita.com/harhogefoo/items/348d89164c9c1bc91f08
- GitHub: https://github.com/harhogefoo/Saizeriya_1000yen
- 概要: 上記に触発されて作られた派生版。任意の予算を設定できる。
- 差分:
  - 1000 円固定ではなく N 円指定
  - Nuxt.js / Vuetify ベース
  - JSON を GitHub Pages にホストして SPA 的に利用
- 示唆:
  - 「任意予算」は需要がある
  - メニューデータを静的 JSON として持つ設計は十分成立する

3. 吉野家1000円ガチャ
- 参考言及: https://qiita.com/marusho_summers/items/94cc865936cbf6b2de1b
- 概要: Qiita 記事内で参考例として挙げられている類似作。
- 備考:
  - 調査時点では元記事や稼働 URL の一次情報確認は弱い
  - 存在は高確度だが、現存サービスとしての継続稼働は未確認

4. マクドナルド公式 栄養バランスチェック
- URL: https://www.mcdonalds.co.jp/app/menu_information/
- 概要: 商品を選んで栄養バランスや情報確認ができる公式ツール。
- 特徴:
  - 公式データで信頼性が高い
  - 複数商品の合算確認ができる
  - アレルギーや商品情報と併用できる
- 弱み:
  - 予算内最適化ではない
  - 条件を満たす候補を探索して提案する機能ではない

### 1-3. OSS / 公開データセット調査

- 出典: https://github.com/marushosummers/Saizeriya_1000yen_pwa

- URL: https://www.kaggle.com/datasets/priyanshusethi/mcdonalds-nutrition-dataset
- 内容: マクドナルドのメニュー栄養情報データセット
- 評価:
  - 海外向け分析用データとしては使える
  - 日本マクドナルドの商品体系と一致しない可能性が高い

#### 調査時点で見つけにくかったもの

- サイゼリヤ・吉野家・松屋・日本マクドナルドを横断した、保守されている OSS データセット
- 日本チェーン向けに価格と栄養を同時に持ち、定期更新されている公開リポジトリ

#### 実務上の結論

- 日本チェーン向けには「公開 OSS データセットをそのまま使う」より、
  公式サイトの公開情報をもとに自前の正規化データを持つ方が現実的。

- サイゼリヤは公式メニューブックはあるが、価格と栄養を機械可読に横断取得しにくい。既存 OSS を叩き台にしつつ手動メンテ前提で考えるのが安全。

## 2. 参考サイトの技術調査

### 2-1. 技術スタック

調査対象:
- サイト: https://saizeriya-1000yen.marusho.io/
- GitHub: https://github.com/marushosummers/Saizeriya_1000yen_pwa
- Qiita(PWA 化記事): https://qiita.com/marusho_summers/items/94cc865936cbf6b2de1b
- Qiita(初版記事): https://qiita.com/marusho_summers/items/a2d3681fac863734ec8a

確認できた事実:

1. 初版は Flask + SQLAlchemy(SQLite) + Heroku + LINE Bot
- 根拠: 2018-09-22 の Qiita 記事

2. 現行公開版はフロントエンド単体の PWA
- 根拠: 2020-12-10 の Qiita 記事
- 構成:
  - `index.html`
  - `gacha.js`
  - `menu.json.js`
  - `service-worker.js`
  - `manifest.json`
  - 静的ホスティングは GitHub Pages

3. データ処理に AlaSQL を使用
- 根拠: Qiita 記事に `alasql(sql, [MENU, budget])` の記述あり
- 用途:
  - JSON を SQL 風に扱う
  - 予算以下メニューの候補抽出

### 2-2. ソースコード公開状況

- GitHub リポジトリは公開されている
  - https://github.com/marushosummers/Saizeriya_1000yen_pwa
- リポジトリ上で確認できるファイル名から、かなり素朴な静的アプリ構成と判断できる

### 2-3. 組み合わせ探索アルゴリズム

#### 確認できたこと

- Qiita 記事には以下の処理断片がある
  - `SELECT * FROM ? d WHERE d.price <= ?`
  - まず「予算以下の単品候補」を抽出している

#### 推測

- 参考サイトの挙動と「ガチャ」という UI から、厳密最適化よりも以下のいずれかの可能性が高い
  1. 予算以下の候補集合からランダムに追加し、予算超過で止める単純ランダム法
  2. カテゴリ制約つきのランダム選択
  3. 一定件数までの組み合わせを列挙してランダム表示

- 少なくとも公開記事からは、典型的な動的計画法ベースのナップサック最適化を前面に出している形跡は弱い。
- 元の目的が「最適解を返す」より「面白い組み合わせを引く」であるため、アルゴリズムは軽量ランダム寄りだった可能性が高い。

#### 今回アプリへの示唆

- 参考サイトと同じ軽さは活かしつつ、今回の要件では以下が必要になる
  - 価格だけでなくカロリー上限、タンパク質下限など複数制約を同時に扱う
  - ランダム生成だけでなく、条件に合う候補を複数返す
  - 「おすすめ順」や「高タンパク順」などソート軸も必要

よって、参考サイトの実装思想は UX 参考にはなるが、探索ロジックは作り直す前提が妥当。

## 3. 今回作るアプリの要件整理

### 3-1. 機能要件

必須:

1. チェーン店を選べる
- 初期対象: サイゼリヤ / マクドナルド / 吉野家 / 松屋

2. 条件を自由に設定できる
- 予算上限
- カロリー上限
- タンパク質下限
- 必要なら脂質上限 / 炭水化物上限 / 塩分上限も将来拡張

3. 条件に合うメニュー組み合わせを表示できる
- 複数候補
- 並び替え可能

4. 合計値を表示できる
- 合計価格
- 合計カロリー
- 合計タンパク質

5. 個別メニュー内訳を確認できる
- 商品名
- 単価
- カロリー
- タンパク質

準必須:

6. 条件未達時のフィードバック
- 「該当なし」だけで終わらず、どの条件が厳しいか示す

7. データ更新日時の表示
- チェーン別に最終確認日を持つ

### 3-2. 非機能要件

1. 高速性
- 条件変更ごとに 1 秒以内で候補更新したい

2. 透明性
- データ出典、更新日、注意書きを見せる

3. 保守性
- チェーン追加時にデータ差し替えだけで済む構造

4. 法務リスク低減
- 非公式であることを明記
- 公式ロゴや画像の扱いに注意

## 4. 技術スタック提案

### 4-1. フロントエンド

結論:
- 第一候補: `Next.js(App Router) + TypeScript`
- 第二候補: `Vite + React + TypeScript`
- 今回は `Next.js` を推奨

理由:

1. Next.js を推す理由
- 将来、API Route やサーバー側探索を足しやすい
- 静的配信から始めても拡張しやすい
- Vercel へのデプロイが簡単
- SEO や OGP を後から足しやすい

2. Vite + React が向くケース
- 初期 MVP を最速で出したい
- 完全クライアントサイドで割り切る
- 将来の SSR 要件が薄い

3. Remix は今回の優先候補ではない
- 強みはあるが、本件では探索ロジックとデータ正規化が主課題
- 初期速度と周辺情報量では Next.js / Vite が有利

### 4-2. 探索アルゴリズム

結論:
- MVP は `枝刈り付き深さ優先探索(Backtracking)` を推奨
- 将来拡張で `多目的スコアリング + 上位K件抽出` に進める

理由:

1. 単純な完全探索
- 利点: 正確
- 欠点: 品目数が増えるとすぐ重い

2. 0/1 ナップサック
- 利点: 価格だけなら適する
- 欠点:
  - 今回は価格だけでなく、カロリー上限とタンパク質下限など複数制約
  - 同一商品を選べるか、サイズ違いをどう扱うかで設計が複雑
- 結論: 単独採用はやや噛み合わない

3. 枝刈り付き DFS / Backtracking
- 利点:
  - 複数制約を自然に扱える
  - 候補上位 N 件だけ集めればよい用途に合う
  - チェーンごとのカテゴリ制約を入れやすい
- 枝刈り例:
  - 価格上限超過時点で打ち切り
  - 残り候補を全部足してもタンパク質下限に届かないなら打ち切り
  - ソート順を「タンパク質効率」「価格効率」にして探索効率改善

4. ビームサーチ
- 利点: 大規模化しやすい
- 欠点: 最適性保証が弱い
- 位置づけ: 品目が数百を超えたら検討

実装方針:

- MVP:
  - 1 チェーンあたり対象を 30-80 品程度に絞る
  - カテゴリ重複制約を必要に応じて追加
  - 上位 20 件だけ返す

- 将来:
  - 目的関数を導入
  - 例: `score = protein * 2 - calories * 0.003 - price * 0.001`
  - 条件を満たす中でスコア上位を返す

### 4-3. データ管理

結論:
- 初期は `正規化 JSON + Zod バリデーション`
- 更新作業が増えたら `SQLite` に移行

推奨スキーマ:

```ts
type MenuItem = {
  id: string
  chain: "saizeriya" | "mcdonalds" | "yoshinoya" | "matsuya"
  name: string
  category: string
  price: number
  calories: number | null
  protein: number | null
  fat: number | null
  carbs: number | null
  sodium: number | null
  sourceUrl: string
  sourceUpdatedAt: string | null
  available: boolean
}
```

理由:

- JSON:
  - Git 管理しやすい
  - 静的配信しやすい
  - 初期開発が速い

- SQLite:
  - 店舗別価格差、期間限定、監査ログを持ち始めると有利

外部 API について:

- 調査時点で、日本チェーン横断でそのまま使える安定 API は見つけにくい
- MVP で外部 API 依存は避けるべき

### 4-4. デプロイ

推奨:

1. 第一候補: Vercel
- Next.js と相性がよい
- 静的でも SSR でも対応可能

2. 第二候補: Cloudflare Pages
- 軽量、エッジ寄り
- 将来 Workers で探索 API を載せやすい

3. GitHub Pages
- 完全静的なら十分
- ただし将来のサーバー探索や管理画面追加には弱い

### 5-2. コンポーネント設計

1. `ConstraintForm`
- 予算、カロリー上限、タンパク質下限、チェーン選択

2. `ResultList`
- 条件一致候補を一覧表示

3. `ResultCard`
- メニュー内訳と合計値、タグ表示

4. `SummaryBar`
- 検索件数、現在の並び順、データ更新日

### 5-3. UX 改善案

差別化しやすい点:

1. 「該当なし」時の説明
- 例: 「予算 700 円以内かつタンパク質 35g 以上は、このチェーンでは 0 件です」
- 代替提案:
  - 予算を +100 円すると 4 件
  - タンパク質条件を 30g にすると 7 件

2. おすすめプリセット
- `高タンパク`
- `低カロリー`
- `コスパ重視`
- `晩ごはん向け`

3. 比較 UI
- 候補を 2-3 件並べて比較

4. 共有しやすい URL
- クエリパラメータに条件を保存

5. データ信頼性の明示
- 各チェーンごとに「出典 URL」「最終確認日」を表示

### 5-4. MVP 定義

最低限作るべきもの:

1. 対応チェーンは 2 つまで
- 例: サイゼリヤ + 松屋
- 理由: データ整備コストを下げる

2. 条件は 3 つに絞る
- 予算上限
- カロリー上限
- タンパク質下限

3. 結果は上位 10-20 件表示
- 合計価格 / カロリー / タンパク質
- 内訳表示

4. データは JSON 手管理
- 自動収集は後回し

MVP でやらない方がよいもの:

- ログイン
- お気に入り同期
- 店舗別価格差対応
- 複雑な機械学習レコメンド
- 自動スクレイピングの本格化

## 6. 推奨ロードマップ

### Phase 1

- 2 チェーン対応
- JSON データ整備
- 条件検索
- 上位候補表示

### Phase 2

- 4 チェーン対応
- URL 共有
- 並び替え
- 「条件を少し緩めた候補」提案

### Phase 3

- データ更新フロー整備
- 比較 UI
- 期間限定メニュー対応
- API / 管理画面化

## 7. 最終提案

### 推奨アーキテクチャ

- フロント: Next.js + TypeScript
- データ: 正規化 JSON
- 探索: 枝刈り付き DFS / Backtracking
- デプロイ: Vercel

### この構成が合う理由

- MVP を速く出せる
- 将来の多チェーン化に耐える
- 複数制約探索に素直
- 静的運用から API 化まで段階的に伸ばせる
